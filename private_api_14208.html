Communication Units points

Endpoints to store measurements (points) for communication units.

/points/communicationUnits post

post

Posts in measurements. Note that measurements could come from many CUs.

We post data to our internal data stores fairly efficiently, so large requests are perfectly fine.

The data points will be processed in a streamed way, by partitioning them into batches of 1000. Each batch will be persisted atomically before continuing with the next one. In case of failure at one of them, the returned error message will indicate the number of successfully stored data points.

The API will automatically find and fill in relevant reference data, like machine, model, etc for the communication units provided. These cannot be overriden.

Data submitted through this endpoints can be seen by all versions of /measurements or /timeseries endpoints, but isn't considered for events or data pump processing.

Communication Units locations

Endpoints to store geo locations for communication units.

/locations/communicationUnits post

post

Posts in locations. Note that locations could come from many CUs.

We post data to our internal data stores fairly efficiently, so large requests are perfectly fine.

The data points will be processed in a streamed way, by partitioning them into batches of 1000. Each batch will be persisted atomically before continuing with the next one. In case of failure at one of them, the returned error message will indicate the number of successfully stored data points.

The API will automatically find and fill in relevant reference data, like machine, model, etc for the communication units provided. These cannot be overriden.

Handling of the speed value looks the same as inside the PWF: it is stored as an independent signal ("proemion.gps.speed") and can only be correlated back to its geo location by timestamp-matching.

Data submitted through this endpoint can be seen by all regular timeseries-serving endpoints, but is not considered for events or data pump processing.

HTTP status code 202

The points are accepted for ingestion.

HTTP status code 400

Received if:

  • a value is not-parseable (e.g. a string given to a number field);
  • a required field is missing;
  • a geo value is outside of its valid range (e.g. latitude greater than 90.0);
  • any other value is outside of its sanity range (e.g. speed at millions of kilometers per hour);

HTTP status code 403

Received if the authenticated subject lacks the "service.dp.points.post" permission.

Machines points

Endpoints to store measurements (points) for machines.

/points/machines post

post

Posts in measurements. Note that measurements could come from many machines.

Measurements will be stored into the machines's CU, so only machines with a single CU are allowed.

We post data to our internal data stores fairly efficiently, so large requests are perfectly fine.

Requests over 10MB will be split up at the backend.

The API will automatically find and fill in relevant reference data for the machines provided.

Data submitted through this endpoints can be seen by all versions of /measurements or /timeseries endpoints, but isn't considered for events or data pump processing.

Errors

Error response structure.

Each error response has the following structure.

{
"code": 404,
"message": "Requested resource not found",
"details": "See http://docs.proemion.com",
"logid": "2342221-321df-342fds23"
}
  • code: HTTP response code.
  • message: Error message.
  • details: Error details.
  • logid: A unique alpha-numeric identifier to track an error event in our logs.

Common error responses

400 (Bad request)

Cause: Invalid parameters in the request. For example, when a String is passed for an Integer/Long parameter. Also caused in situations where the request is syntactically correct but the server could not process it as the semantics are invalid. For example, a non-existent database field is being requested.

Examples:

{
"code": 400,
"message": "Cannot cast '12acd' to type class java.lang.Long",
"details": "See http://docs.proemion.com"
}
{
  "code": 400,
  "message": "field [queries] not found",
  "details": "See http://docs.proemion.com",
  "logid": "3971321-a32df-762fde28"
}

#### 403 (Forbidden) *Cause*: Insufficient permissions for the current user to access a given resource.

Example:

{
  "code": 403,
  "message": "Not entitled for this request, please contact customer support",
  "details": "See http://docs.proemion.com",
  "logid": "4171451-bd1df-798fde23"
}

Authentication

Endpoint to authenticate the user and obtain an access token for accessing any entitled enpoints. This follows the OAuth 2.0 password based grant. Too many consecutive failed authentication attempts - or too frequent attempts in general - may lead to temporary lock-out of the user. Please see the description of the HTTP 429 response.

/authz/oauth2/token post

post

Obtain an access token for a given user and client. The response code, either 200 or 303, depends on whether the redirect_uri optional field was not set or set in the request, respectively.

post /authz/oauth2/token

Obtain an access token for a given user and client. The response code, either 200 or 303, depends on whether the redirect_uri optional field was not set or set in the request, respectively.

  • Response

Clustering

Endpoints to obtain clustering reports.

/clustering/kmeans post

post

Generate a k-means clustering report. Each report point (aggregated point) corresponds to a machine of the model. Aggregated points are calculated by applying the reduction function to all measurements of given signal of the machine. The number of requested clusters has to be greater than the number of machines of a model.

post /clustering/kmeans

Generate a k-means clustering report. Each report point (aggregated point) corresponds to a machine of the model. Aggregated points are calculated by applying the reduction function to all measurements of given signal of the machine. The number of requested clusters has to be greater than the number of machines of a model.

Secured by OAuth 2.0

All entitled endpoints require a valid token to access the resource.

  • Request
  • Response
  • Security

Body

Media type: application/json

Type: object

Properties
  • signal: required(string)

    the signal name to query.

  • model: required(string)

    the modelid; all machines of the model will be used for clustering.

  • reductionFunction: required(string)

    Reduce the timeseries for each machine to a single point. Each aggregated point representing a machine will be used to cluster the machines. Supported reduction functions are:

    • min - smallest value,
    • max - largest value,
    • average - mean or average value,
    • std - standard deviation,
    • sum - sum of the values.
  • clusters: required(integer)

    the number of clusters to seach for using aggregated points. clusters has to be less or equal to the number of machines in the model.

  • from: required(integer)

    Milliseconds since epoch. Denotes starting point in time for aggregation. Maximal time range is 30 Days.

  • to: required(integer)

    Milliseconds since epoch. Denotes end point in time for aggregation. Maximal time range is 30 Days.

  "signal" : "value.oil.temperature",
  "model": "xyzq9876",
  "reductionFunction": "average",
  "clusters": 3,
  "from": 1451602800000,
  "to": 1451602900000
}

Machines

Endpoints to retrieve information about machine(s).

/machines get

get

Get a list of machines satisfying the given criteria.

get /machines

Get a list of machines satisfying the given criteria.

Secured by OAuth 2.0

All entitled endpoints require a valid token to access the resource.

  • Request
  • Response
  • Security

Headers

  • Accept-Language: (string)

    Desired language for the response. Both language and locale codes are acceptable (en, en-US). If no header specified or locale is unknown then default en locale is applied.

Query Parameters

  • q: (string)

    The query to execute for filtering. Follows the RSQL syntax with the following limitations:

    • multi-character wildcards (asterisk: *) cannot appear in the middle of a search string;

    Examples:

    filterByName:

    name==*lift*

    filterByModelId:

    modelEntity.id==76
  • modelname: (string)

    The model name of machine(s). If this is present then, the q parameter will be ignored.

  • geo: (boolean)

    If true, includes the last geo point reported by the machines.

  • signal: (array of )

    The keys of signals to include the latest measurements for. See /measurements for more details. We return latest values for the supplied signals.

  • status: (boolean)

    If true, includes the online status of the machine.

  • contracts: (boolean)

    If true, contract information is included for every machine. To access this information the user needs permission to view contracts.

  • dataConfiguration: (boolean)

    If true, data configuration summary is included for every machine.

  • offset: (integer - default: 0)

    The number of results to skip from start of the result set. Can be empty or has to be populated together with limit.

  • limit: (integer - default: 20)

    The number of results to return after offset. Maximum 1000. Can be empty or has to be populated together with offset.

  • sort: (string)

    Ascending by default. The name of the property we want to use for sorting. Substract sign (-) may be used before the name to indicate descending direction.

Type: array of object

  • serial: required(string)

  • pin: required(union of string or nil)

  • organization: required(object)

    The organization the machine belongs to.

    • id: required(string)
    • name: required(string)
  • communicationUnits: (array of object)

    Communication units linked to this machine.

    Items: items

    • id: required(string)
    • name: required(string)
    • pdc: (object)

      This attribute is deprecated and is going to be removed by the end of September 2019. Please refer to the newly-added dataConfiguration attribute of the machine.

      • uuid: (string)
      • version: (string)
  • model: (object)

    The model of the machine.

    • id: required(string)
    • name: required(string)
  • dataConfiguration: (object)

    The data configuration resolved for this machine, if any. Only visible if the data configuration itself is accessible to the requester.

    This item needs to be explicitly requested.

    • id: required(string)
    • version: required(string)
    • name: required(string)
  • dataConfigurationResolution: (one of model, machine, none)

    The indicator of the data configuration's source. Visible even when the data configuration itself is not accessible to the requester. The possible values are:

    • model: if the data configuration is assigned at the model level and not overriden by this machine.
    • machine: if the data configuration is assigned explicitly to this machine, regardless of the model's data configuration.
    • none: if neither this machine nor its model has any data configuration assigned.

    This item needs to be explicitly requested.

  • measurements: (object)

    The latest signal values of the machine.

    This item will only be present if one or more signals are explicitly requested.

    • numeric: required(object)

      Numeric signal values.

      • //: required(object)

        The measurement of a numeric signal indicated by the property name.

        • time: required(integer)

          The time of measurement, in epoch milliseconds.

        • value: required(number)

          The measured value at the given time.

    • string: required(object)

      Textual signal values.

      • //: required(object)

        The measurement of a textual signal indicated by the property name.

        • time: required(integer)

          The time of measurement, in epoch milliseconds.

        • value: required(string)

          The measured value at the given time.

  • geo: (object)

    Last known position of the machine. This item needs to be explicitly requested.

    • time: required(integer)

      Milliseconds since epoch. Denotes when the position has been reported by the machine.

    • longitude: required(number)
    • latitude: required(number)
  • status: (string)

    Provides the current connection status for the machine. It can be one of realtime, online, offline.

    • If at least one communication unit associated with the machine is connected to in realtime mode, then the machine status is realtime.
    • If not, but at least one is online, then the machine status is online.
    • If not the machine is considered offline.

    This item has to be explicitly requested. It can also be absent if the machine has no registered communication units.

  • contract: required(object)

    Contract details for this machine.

    Contract details have to be explicitly requested through query parameters.

    If we failed to get the contract details for this machine we'll exclude this attribute entirely.

    Possible failure reasons:

    • requesting user doesn't have the required permissions
    • machine missing contract details
    • temporary internal datasource problems
    • contractId: (string)

      Contract activation time in millis. To access this information the user needs permission to manage contracts.

    • startDate: (integer)

      Contract start time in millis. To access this information the user needs permission to manage contracts.

    • tariffs: (array of object)

      Tariffs applied to this contract

      Items: items

      • description: required(string)

        Short tariff description

      • nextBillingDate: (integer)

        Next billing date for this tariff. To access this information the user needs permission to manage contracts. Millis since epoch.

      • fromDate: (integer)

        Tariff validity start date. To access this information the user needs permission to manage contracts. Millis since epoch.

      • toDate: (integer)

        Tariff validity end date. To access this information the user needs permission to manage contracts. Millis since epoch.

Timeseries

Endpoints to query timeseries. Timeseries are composed of raw or aggregated values for a particular signal of a machine or a model.

Timeseries are useful when querying a number of values over a date range, we support a wide range of aggregations and raw (non-aggregated) datapoints.

An example usage is providing data for plotting signal value changes over time.

This is in contrast to the /measurements API, which returns just the last data point.

/timeseries post

post

Run the given query.

IMPORTANT NOTE: As a result of our internal bucket division and how aggregation operations are calculated, you might experience an unexpected amount of results in those situations where aggregation functions are used in combination with a time range that crosses more than one of our internal data buckets. In those situations, the result will contain one datapoint per crossed bucket. As an example, a request with a selected bucket size of 1h under a 2h time range could be answered with 2+ datapoint. Assume, storage contains data points with timestamps "10:30:00", "11:30:00", "12:03:00". If requested timerange is "Nov 21 2018 10:05:00" - "Nov 21 2018 12:05:00" then the response will contain 3 buckets (instead of 2) with timestamps "10:00:00", "11:00:00" and "12:00:00". Keep in mind that the bucket content is not wrong. The combined 3 buckets get the correct number. For example, if you use the sum aggregation, the total for the 3 buckets will be as expected. This is just the way the response is presented (in a full hour interval broken on the hour).
